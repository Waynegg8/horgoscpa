## 📋 文檔長度標準

為了確保 AI 和開發人員能夠有效閱讀和理解，所有文檔都遵循長度限制：

| 文檔類型 | 建議行數 | 最大行數 |
|----------|----------|----------|
| 規範文檔 | 80-120 | 150 |
| 範例文檔 | 60-100 | 120 |
| 索引文檔 | 50-80 | 100 |

**超過限制時：** 必須拆分成多個小文檔，每個專注於單一主題。

## 🔍 使用建議

### 給 AI 的建議

**開始開發任務時：**
1. 閱讀 [專案開發規格總覽](../AI開發指南/專案開發規格總覽.md) - 了解整體規範
2. 根據任務類型，閱讀相關的規範文檔（通常 2-3 個文檔即可）
3. 開始開發時參考規範

**不要一次讀取所有規範：**
- ❌ 不要試圖一次讀取整個目錄
- ✅ 根據需要讀取特定文檔
- ✅ 使用快速查找指南找到相關文檔

### 給開發人員的建議

**新加入專案：**
1. 先閱讀本 README（了解規範結構）
2. 根據你的角色選擇：
   - 後端開發：閱讀「後端開發規範」
   - 前端開發：閱讀「前端開發規範」
   - 全端開發：兩者都讀
3. 需要時查閱特定規範

**日常開發：**
- 使用「快速查找指南」快速找到相關規範
- 遇到疑問時查閱對應文檔
- 不確定時參考現有代碼的實作方式

## ⚠️ 重要提醒

### 絕對禁止修改外部網站

**警告文檔：** [🚨外部網站禁止修改清單](../🚨外部網站禁止修改清單.md)

**禁止修改：**
- ❌ 根目錄的所有 `.html` 檔案
- ❌ `blog/` 目錄
- ❌ `services/` 目錄
- ❌ `assets/` 目錄
- ❌ `content/` 目錄

**允許修改：**
- ✅ `timesheet-api/` 目錄（內部系統後端）
- ✅ 未來的 `admin/` 或 `internal/` 目錄

## 📝 規範更新流程

當需要更新規範時：

1. 更新對應的規範文檔
2. 如果影響其他文檔，同步更新
3. 更新本 README 的「最後更新」日期
4. 在相關文檔底部更新版本號和變更記錄
